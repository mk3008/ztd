# ztd-cli to @ashiba/cli Code Migration Plan

## Summary

Ashiba migrates useful `ztd-cli` behavior into `@ashiba/cli` while preserving source behavior before redesign. `ztd-cli` must not be deleted, and Ashiba must not introduce an ORM runtime.

Ashiba may depend on `rawsql-ts` core through npm for SQL AST analysis. The migration target is the user-facing `ztd-cli`, sqlgrep, and migration workflow surface, not a fork of the full parser internals.

## Stage 1: Package Skeleton

Create:

- `package.json`
- `pnpm-workspace.yaml`
- `packages/cli/package.json`
- `packages/cli/tsconfig.json`
- `packages/cli/src/index.ts`
- `packages/cli/tests/smoke.test.ts`
- `packages/cli/README.md`

Target package: `@ashiba/cli`
Target bin: `ashiba`

## Stage 2: Command Framework

Migrate command registration from `packages/ztd-cli/src/index.ts`.

Rename:

```txt
program.name("ztd") -> program.name("ashiba")
```

Preserve behavior first. Do not deeply redesign commands in this stage.

`ashiba init` may create README files and ordinary project documentation, but Ashiba must not distribute AI behavior files such as `AGENTS.md`, `AGENT.md`, `SKILL.md`, Codex skills, prompts, or agent policy bundles. Those files change tool behavior and are outside Ashiba distribution. Use scaffold structure, reviewable contracts, command descriptors, and AI-oriented error output to guide AI work instead.

Do not migrate watch-mode automatic regeneration for DDL-derived schema model artifacts. Ashiba should prefer explicit drift detection and actionable error messages over background rewrites. A stale generated artifact should fail a check or test with cause and next action, then be regenerated by an explicit command.

## Stage 3: DDL and Migration Commands

Migrate and reshape:

- `ddl migration generate`
- `ddl migration info`
- migration DDL generation from two explicit DDL inputs
- migration risk/info output from two explicit DDL inputs

Keep migration artifacts review-oriented. Do not add automatic apply behavior.

Migration query generation is in Ashiba CLI scope: compare two DDL inputs, generate migration DDL, and emit risk/info output. DB connection and migration apply execution are not CLI responsibilities.

## Stage 4: Feature and Query Scaffolding

Migrate:

- `feature scaffold`
- `feature query scaffold`
- `feature generated-mapper generate`
- `feature generated-mapper check`

Preserve visible SQL files, query boundary, feature boundary, generated row mapper, and drift checks.

## Stage 5: Test Scaffolding

Migrate:

- `feature tests scaffold`
- ZTD lane if retained
- traditional DB-backed lane if retained
- mapper test preference for the ZTD lane
- performance test preference for the traditional DB-backed lane

If ZTD remains, document it as a test technique, not the product identity.

## Stage 6: Query Analysis and sqlgrep

Migrate:

- `query uses table`
- `query uses column`
- `query match-observed`
- `query outline`
- `query graph`
- `query slice`
- `query plan`
- `query lint`

Public v1 should migrate the full `ztd-cli` query analysis command set. Keep `sqlgrep` as the capability name and expose it through Ashiba query commands where useful.

Use `rawsql-ts` core AST APIs for structural SQL parsing and analysis instead of reimplementing parser internals inside Ashiba.

## Stage 7: model-gen

Migrate or adapt `model-gen`. Preserve `QuerySpec` id, `sqlFile` metadata, named params, and output mapping.

Temporary mapping:

```txt
ztd model-gen -> ashiba model-gen
```

## Stage 8: Driver Adapter Packages

Create or migrate:

- `@ashiba/driver-adapter-core`
- `@ashiba/driver-adapter-pg`

The requested source path `packages/drivers/driver-adapter-core` was not present locally, so the first core package may need to be designed from adapter-node-pg evidence. Named-parameter metadata generation belongs inside `@ashiba/cli`; runtime adapters consume precomputed metadata instead of depending on a standalone binder package.

Transaction management is not an Ashiba responsibility. Feature scaffolding may emit transaction-shaped starter code based on the `ztd-cli` baseline where useful, but after generation that code is editable application code.

## Stage 9: Rename Terminology

Classify terms before renaming:

- Product-level ztd terms: rename to Ashiba.
- Test-technique ZTD terms: keep only when they mean Zero Table Dependency testing.
- rawsql-ts terms: keep only when referring to the source project or dependency.
- RFBA terms: keep where the architecture concept remains useful.

## Decided Direction

- Public v1 query analysis migrates the full `ztd-cli` query analysis command set.
- Unused named parameters are errors.
- Initial database target is PostgreSQL; MySQL and SQL Server are target families for later driver adapters.
- PostgreSQL `ddl pull` through `pg_dump` is implemented as the `@ashiba/ddl-pull-pg-dump` helper package because of `pg_dump` references, not as an initial migration query generation responsibility or `pg` driver adapter responsibility. The package name is not generic because it does not cover MySQL or SQL Server DDL pull.
- Keep `sqlgrep` as the capability name.
- `rawsql-ts` core AST analysis remains an allowed npm dependency for Ashiba tooling packages.

## Risk List

- Over-renaming could destabilize the baseline.
- Under-renaming could leave Ashiba with unclear product identity.
- Testkit adapters could be mistaken for production driver adapters.
- Logger-ready events could be confused with an actual logger.
- Migration artifacts must stay review-only.

## Verification Steps

Each migration stage must support:

- `pnpm install`
- `pnpm build`
- `pnpm test`
- `pnpm pack`
- isolated consumer install smoke test

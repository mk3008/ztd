# Ashiba Concept Overview

Show me the SQL.

Ashiba handles the boring parts.

## Position

Ashiba is a rebrand of `rawsql-ts/packages/ztd-cli`.

Ashiba is a SQL scaffolding product for TypeScript applications. It keeps SQL visible and scaffolds the repetitive work around it. Its Ashiba Runtime Zero promise applies to application code generated by `@ashiba/cli`, not to every Ashiba package.

Ashiba is not PostgreSQL-only. The first starter and first production adapter may use PostgreSQL, but product-level concepts, configuration names, and test environment names should stay DBMS-neutral unless the artifact is explicitly PostgreSQL-specific.

Scaffolded output is not locked or hidden behind `generate`. Humans and AI agents may edit generated SQL-adjacent code after scaffolding, and that code remains under drift checks.

SQL must stay SQL. Query logic should be directly runnable in a SQL client for debugging, without translating a query DSL back into SQL.

Source SQL must also remain ordinary SQL. Ashiba should not require proprietary comments, replacement markers, template directives, or other Ashiba-only syntax inside SQL files.

SQL must also stay maintainable. Source SQL uses named parameters, such as `:name` or `@name`, and the DB driver wrapper translates them to the concrete driver's placeholder format.

The `@ashiba/cli` Runtime Zero path should not introduce hidden dynamic SQL rewriting into generated application code. SQL should remain directly understandable and debuggable. Driver adapters and extension packages are separate responsibility areas, but Ashiba-specific control data should live in explicit generated metadata files or query models rather than in proprietary SQL syntax.

Type safety is guaranteed through mapper tests and DB-backed integration tests, not runtime type checking of every DB result row. The database already enforces strong type constraints, and Ashiba verifies the SQL-to-mapper contract during development.

Unit tests have two lanes: traditional tests and Zero Table Dependency tests. Mapper tests should prefer Zero Table Dependency; performance tests should prefer traditional DB-backed tests.
Performance tuning sessions collect representative row-count requirements, execution-plan evidence, timings, timeout status, and AI/human next actions. Candidate indexes are sandbox-only until explicitly promoted into `db/ddl`.

## Core Boundary

Ashiba Runtime Zero means `@ashiba/cli` generates native TypeScript application code, such as DTO definitions and mappers, and that generated application code does not need Ashiba CLI/runtime libraries at runtime.

This is important because deployed applications avoid Ashiba CLI runtime library dependency, avoid forced security updates for an unused runtime layer, and can patch generated code directly when needed.

Driver adapters and extension packages may have runtime dependencies. The Ashiba Runtime Zero claim does not forbid those packages from existing.

`@ashiba/cli` should still provide practical development support often expected from ORMs, excluding database drivers. It provides that support through scaffolding, generated code, tests, drift checks, migration review, and query analysis, not through runtime relation loading, lazy loading, unit-of-work tracking, or entity ownership.

It does not mean:

- no database driver
- no driver adapter
- no extension runtime
- no connection pool
- no application-owned transaction handling

Initial production driver work starts with PostgreSQL. MySQL and SQL Server are target database families for later driver adapters. PostgreSQL-specific package names are appropriate only when the package wraps a PostgreSQL-specific driver or tool.

## What Ashiba Owns

- Scaffolding
- Generated code layout
- Native generated DTO definitions and mappers
- Generated code that stays visible and may be edited by humans and AI agents
- Library-owned generated-folder unit-test schema files derived from DDL
- Test scaffolding
- Test lanes for traditional and Zero Table Dependency tests
- Performance tuning sessions as manual traditional DB-backed evidence loops
- Drift detection
- Mapper-tested type safety
- Migration artifacts for review
- Migration query generation from two DDL inputs
- SQL search
- SQL impact analysis
- Named parameter compilation
- Safe sort profile support
- Logger-ready event seams
- Human-oriented and AI-oriented error output modes
- JSDoc for public exported functions
- CLI help surfaces, with AI-oriented help when useful
- Dry-run or equivalent preview for mutating CLI commands

## What Applications Own

- Business requirements
- Business SQL intent
- Transaction boundaries
- Connection lifecycle
- Deployment
- Runtime logger implementation
- Production log security policy

## Package Responsibility Categories

Ashiba is a multi-package product. Review concepts by package responsibility:

- Repository Philosophy: product promise, visible SQL, Ashiba Runtime Zero for CLI-generated application code, and no ORM/query DSL boundaries.
- CLI: `@ashiba/cli`, scaffolding, generated review artifacts, tests, drift detection, migration review, sqlgrep, and impact analysis.
- Driver Packages: `@ashiba/sql-*`, `@ashiba/driver-adapter-*`, `@ashiba/testkit-adapter-*`, and `@ashiba/dialect-*`.
- Extension Packages: `@ashiba/sql-transform-*`, DDL pull extensions, and future `@ashiba/extension-*` packages.

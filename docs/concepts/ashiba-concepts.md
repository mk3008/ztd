# Ashiba Concepts

This is the initial ConceptSpec source for Ashiba. Ashiba is a rebrand of `rawsql-ts/packages/ztd-cli`, and these concepts define the target product boundary before code migration starts.

## Status Vocabulary

- `done`: implemented in the ztd-cli baseline and expected to carry forward.
- `mostly done`: implemented enough to reuse with limited rename or packaging work.
- `partial`: present but incomplete for the Ashiba concept.
- `provisional`: kept as an idea, but must be redesigned before implementation.
- `missing`: not implemented in the baseline.
- `deferred`: intentionally postponed by package scope or implementation pass.
- `needs rename`: implemented under ztd-cli/rawsql-ts naming.
- `needs migration`: implemented but must be moved or repackaged for Ashiba.

## Ashiba

### Definition

Ashiba is a SQL scaffolding product for TypeScript applications. Ashiba keeps SQL visible and scaffolds the boring parts around it. Its Ashiba Runtime Zero promise applies to application code generated by `@ashiba/cli`, not to every Ashiba package.

Ashiba is not a PostgreSQL-only product. The initial starter and first production driver adapter may use PostgreSQL to give users a concrete DB-backed test path, but product concepts, configuration names, environment variable names, and package boundaries should avoid implying that Ashiba is dedicated to one DBMS.

### Why It Exists

Developers and AI agents should be able to review, debug, test, edit, and evolve SQL directly without accepting an ORM runtime layer.

Ashiba Runtime Zero CLI output reduces dependency pressure. Applications do not carry an Ashiba CLI/runtime library for generated application code, so they are not forced to update Ashiba for runtime security advisories that do not affect deployed code. When generated code has a problem, the team can patch its own application code immediately instead of waiting for an upstream library release.

Ashiba should cover practical development support often expected from ORMs, excluding database drivers, through scaffolding, generated code, tests, drift checks, migration review, and query analysis. It should not provide those capabilities through a runtime ORM layer.

### Included Responsibilities

- Product identity for the rebranded `ztd-cli` foundation.
- CLI scaffolding for SQL-first TypeScript projects.
- DBMS-neutral product vocabulary and configuration surfaces, even when the initial starter uses PostgreSQL.
- Explicit DBMS starter selection during setup; Ashiba should not silently choose an application database driver.
- Customer tests per selected DBMS and wrapped driver that prove minimum functionality, not only installability.
- Disposable verification artifacts should live under `tmp/` or another explicit temporary workspace; `src/sql` must not become an Ashiba convention for work files or generated customer-test fixtures.
- Application-owned package manager state. `ashiba init` may scaffold files, but it should not create or manage `package.json` as if Ashiba owned the consuming project.
- Review-oriented generated artifacts around SQL, DDL, tests, mappings, migrations, sqlgrep, and impact analysis.
- Generated code that humans and AI agents may edit after scaffolding.

### Excluded Responsibilities

- ORM runtime behavior.
- Runtime entity tracking.
- Business SQL ownership.
- Application transaction policy.
- Application dependency installation and package manager policy.
- Treating the initial PostgreSQL starter path as proof that Ashiba is PostgreSQL-only.
- Treating tarball install, import smoke, or typecheck alone as a customer test.

## Visible SQL

### Definition

SQL must remain readable, reviewable, editable, executable, and searchable. SQL should not be hidden behind an ORM model layer or query DSL.

SQL files must remain compatible SQL. Ashiba must not require proprietary comments, replacement markers, template directives, or other Ashiba-only syntax inside source SQL. If SQL only works after adding special Ashiba notation, it is no longer SQL-compatible for Ashiba's purposes.

### Why It Exists

When behavior, performance, or production data is questioned, reviewers ask for the SQL. Ashiba starts from that review boundary and keeps the SQL easy to edit after scaffolding.

### Included Responsibilities

- SQL files as first-class assets.
- SQL files and adjacent generated code that are easy for humans and AI agents to modify.
- Named parameters for maintainable SQL.
- SQL search and impact analysis.
- SQL-first generated test and mapper contracts.
- No proprietary Ashiba notation inside source SQL.

### Excluded Responsibilities

- Replacing SQL with application-owned model APIs.
- Hiding SQL inside generated runtime layers.
- Treating generated SQL-adjacent code as opaque or off-limits to human/AI edits.
- Requiring Ashiba-only comments, replacement markers, or template directives for SQL execution.

## Ashiba Runtime Zero

### Definition

Ashiba Runtime Zero means application code generated by `@ashiba/cli` can run without Ashiba CLI or Ashiba runtime libraries. The CLI generates native TypeScript code, such as DTO definitions and mappers, that belongs to the application. This is more precise than the older or looser phrase "runtime-free generated code."

Database drivers, driver adapters, and extension packages are explicitly outside this claim and may have runtime dependencies.

### Why It Exists

`@ashiba/cli` should move safety to generated code, development-time checks, and scaffolded tests instead of adding an Ashiba runtime abstraction.

Avoiding an Ashiba runtime dependency also avoids forced security updates for unused Ashiba runtime code. Generated code is local application code, so teams can inspect, edit, and patch it directly.

The CLI should still provide the development workflow breadth that teams often expect from ORMs, excluding database drivers. It provides this through feature and query scaffolding, DTOs, mappers, query IDs, tests, drift checks, migration review artifacts, and query analysis rather than through relation loading, lazy loading, unit-of-work tracking, or runtime entity ownership.

### Included Responsibilities

- Development-time scaffolding.
- Native generated application code such as DTO definitions and mappers.
- Drift checks.
- Generated tests and readable generated code.

### Excluded Responsibilities

- Claiming applications do not need database drivers.
- Requiring Ashiba CLI or Ashiba runtime libraries in CLI-generated application runtime code.
- Claiming driver adapters or extension packages cannot have runtime dependencies.
- Owning connection pools.
- Owning transaction handling.
- Adding an Ashiba data-access runtime layer.

## CLI Dry Run

### Definition

Ashiba CLI commands that create, overwrite, delete, move, or otherwise mutate files or external state must provide a dry-run mode before the mutating behavior is considered complete.

Dry run means the command reports what it would do without writing files, changing generated artifacts, starting external operations, or applying changes. Read-only analysis commands do not need a dry-run flag because their normal execution is already observational.

### Why It Exists

Ashiba is designed for explicit review and visible broken-state recovery. Humans and AI agents should be able to inspect the planned file changes, migration outputs, generated boundaries, and other effects before allowing the command to mutate the workspace.

This is especially important for scaffolders and migration helpers, because the safe behavior is to preview intent, confirm paths and generated content, and then write explicitly.

### Included Responsibilities

- `--dry-run` or an equivalent preview mode for mutating CLI commands.
- Clear output describing files, paths, generated SQL, or actions that would occur.
- No filesystem mutation in dry-run mode.
- No external state mutation in dry-run mode.
- Tests for dry-run behavior on mutating commands.

### Excluded Responsibilities

- Requiring dry-run on commands that only print help, describe commands, lint, inspect, or report.
- Treating dry-run output as a substitute for human review when the command is later executed for real.

## Tooling AST Dependency Policy

### Definition

Ashiba development tools may depend on `rawsql-ts` core through npm for SQL AST parsing and structural analysis. This policy applies to Ashiba packages such as query analysis, sqlgrep, migration review helpers, and other CLI-side development workflows.

This dependency does not weaken Ashiba Runtime Zero because Runtime Zero is a promise about application code generated by `@ashiba/cli`. Generated application DTOs, mappers, and feature code must not require Ashiba CLI/runtime libraries or `rawsql-ts` parser internals at application runtime.

If a capability is used only at development time and exists to protect the Runtime Zero generated-code workflow, prefer integrating it into `@ashiba/cli` instead of creating a separate package. `sqlgrep` is the representative case: it is a development-time investigation capability, so it belongs in the CLI until a real non-CLI library consumer exists.

### Why It Exists

Ashiba is a rebrand and integration of the `ztd-cli`, sqlgrep, and migration workflow surface. Reimplementing the complete `rawsql-ts` AST parser inside Ashiba would add migration cost and parser drift without improving the product boundary.

Using `rawsql-ts` core keeps SQL analysis AST-first and stable while allowing Ashiba to focus on the product contract: visible SQL, editable generated code, drift checks, migration review, and ORM-like development support without a runtime ORM.

Development-time SQL understanding should prefer tested AST APIs over regular-expression or hand-written lexical parsing. Regex and lexical scans are acceptable only as narrow helpers for non-structural concerns such as generated TypeScript metadata extraction, path normalization, source-offset slicing, or parser-failure diagnostics.

Ashiba should dislike silent fallback. The `rawsql-ts` AST parser is a trusted, heavily tested dependency for development-time SQL understanding. If dev-time SQL structural analysis cannot be expressed through that AST path, first treat it as an Ashiba AST traversal gap or a reportable `rawsql-ts` parser/AST issue, not as permission to silently ship a regex-based interpretation. The tool should fail with a clear cause and next action, report the unsupported shape, or require an explicit human-controlled option before using fallback behavior. Quietly repairing, rewriting, or reinterpreting SQL makes failures harder to understand and is not aligned with Ashiba's explicit drift recovery concept.

### Included Responsibilities

- Use `rawsql-ts` core AST APIs in Ashiba tooling when structural SQL analysis is needed.
- Keep those dependencies in Ashiba tooling packages, not in generated application code.
- Prefer AST-first parsing over ad hoc string parsing for query analysis, sqlgrep, CTE planning, linting, and SSSQL helpers.
- Prefer AST-backed extraction for development-time table, column, result-column, clause, and predicate analysis.
- Treat regex-based SQL structure extraction as parser/AST capability debt unless it is limited to source locations or explicitly reported parser-failure diagnostics.
- Do not keep pre-release migration shims; unsupported shapes should fail clearly or be fixed at the AST/parser boundary.
- Prefer fixing the AST parser, AST traversal, or query model metadata over adding quiet local string parsing.
- Treat unexpected `rawsql-ts` parse failures as bugs to investigate or report, because the AST parser is the intended source of truth for development-time SQL structure.
- Prefer CLI-internal implementation for Runtime Zero support capabilities that are only used during development.
- Document when an Ashiba package is a development/tooling package that may carry parser dependencies.

### Excluded Responsibilities

- Porting or forking the entire `rawsql-ts` AST parser into Ashiba.
- Requiring generated application code to import `rawsql-ts`.
- Splitting development-only CLI capabilities into fine-grained packages without a concrete external consumer.
- Treating AST parser dependency as permission to introduce runtime ORM behavior.
- Hiding SQL behind query DSL or hidden CLI-generated SQL rewriting.
- Using regex as the primary mechanism for development-time SQL structural analysis when an AST API can provide the same information.
- Silently falling back from AST-backed analysis to regex or lexical interpretation for SQL structure.
- Quietly repairing, rewriting, or accepting unsupported SQL shapes without telling the human what was unsupported and what should be fixed.

## Mapper-Tested Type Safety

### Definition

Ashiba should guarantee DB result shape and mapper type safety through mapper tests and DB-backed integration tests, not through runtime type checking of every row. When the database schema is strongly typed and the DB integration tests prove the SQL-to-mapper contract, runtime validation overhead is not part of Ashiba's application runtime model.

### Why It Exists

Runtime zero development still needs safety, but Ashiba should place that safety in generated mapper tests, drift checks, and DB-backed verification. Revalidating DB results at runtime duplicates guarantees already provided by the database and verified mapper contract, and adds overhead to application execution.

### Included Responsibilities

- Generate or scaffold mapper tests that prove SQL result rows map to DTOs correctly.
- Support DB-backed integration tests where the real database type constraints are exercised.
- Require customer functional tests for each supported wrapped driver to execute real DB-backed mapping checks, not only package installation or adapter import checks.
- Treat mapper tests and drift checks as the type-safety gate.
- Prefer Zero Table Dependency for mapper tests.
- Keep generated DTOs and mappers readable and reviewable.

### Excluded Responsibilities

- Runtime type validation of every DB result row.
- Runtime schema parsing for mapper output as an Ashiba requirement.
- Business validation or API input validation.
- Replacing database constraints with application-side type checks.

## No Query DSL Ceremony

### Definition

Ashiba should not require developers to express business SQL through a library-owned query DSL. SQL must remain SQL, and the reviewable SQL should be executable immediately in a SQL client for debugging.

### Why It Exists

SQL-first development should keep SQL as SQL for debugging ease. When a query misbehaves, developers should be able to copy or open the SQL in a normal SQL client and run it without translating a fluent API, model graph, or library DSL back into SQL.

### Included Responsibilities

- SQL-first source files.
- SQL that can be run in a SQL client for debugging.
- Optional SQL transforms that preserve SQL readability.

### Excluded Responsibilities

- Replacing SQL with fluent model APIs.
- Making a DSL the authoritative source that must be translated before debugging.

## CLI No Hidden SQL Rewrite

### Definition

The `@ashiba/cli` Runtime Zero path must not hide dynamic SQL rewriting inside generated application code. SQL generated or scaffolded by the CLI should remain understandable as a whole, directly debuggable, and safe from hidden string-assembly risks.

This concept is not a repository-wide ban on SQL handling. Driver adapters and extension packages have their own boundaries and may provide bounded SQL handling when their package concept requires it.

### Why It Exists

Hidden CLI-generated SQL rewriting makes the full query harder to understand, makes debugging harder, and increases injection risk when user-controlled fragments are mixed into SQL. Ashiba's CLI should prefer visible SQL files, generated static code, and explicit review artifacts.

### Included Responsibilities

- Keep business SQL visible as SQL.
- Avoid hidden CLI-generated rewriting for filters, joins, projections, or business conditions.
- Require generated or scaffolded SQL artifacts to stay reviewable.
- Point bounded driver behavior, such as safe sort rendering, to the driver adapter concept instead of treating it as a CLI exception.

### Excluded Responsibilities

- Dynamic runtime SQL assembly from user input.
- Hidden SQL rewriting inside generated helpers.
- Runtime query planning or business condition rewriting.
- Treating scalar or pipeline transform extensions as part of the core CLI Runtime Zero path.

## File-Backed Runtime SQL

### Definition

Runtime application code should not pass arbitrary SQL strings into Ashiba execution boundaries. Runtime execution should start from a reviewed SQL file or a generated query source object that carries the SQL text, SQL path, and matching CLI-generated query model metadata.

The final database driver call inevitably receives a SQL string. That string is an internal adapter detail, not the public application boundary. The application-facing boundary should be query-model-backed so Ashiba can verify the source hash, named-parameter metadata, and safe-sort metadata before execution.

This runtime boundary is separate from development-time SQL authoring commands. A command such as `ashiba query sssql add` may accept a column name or an EXISTS subquery text as authoring input only as a way to generate or update a SQL file during development. The result must be written back to a SQL file and the generated query metadata must be refreshed. That does not permit runtime execution APIs to accept arbitrary SQL strings.

This concept applies to scaffolded and generated application code, not only to code inside driver adapter library packages. When `@ashiba/cli` generates a SQL client, feature executor, connection seam, or starter adapter file, that generated code is part of the concept check surface. A generated starter must not bypass the file-backed or query-model-backed execution boundary just because the unsafe code lives in the consuming project after scaffolding.

### Why It Exists

If Ashiba accepts an arbitrary SQL string at runtime, the adapter cannot know whether the string came from a reviewed file, a generated boundary, concatenated application logic, or user-controlled input. Requiring a file-backed query source closes that injection route at the Ashiba boundary. If a developer intentionally writes a SQL file or generated boundary that embeds unsafe SQL, that is outside Ashiba's automatic protection: the developer has already crossed the source-code trust boundary.

Safe sort is the narrow exception to fully static SQL text. It is not a free-form SQL string escape hatch. The requested sort key must exactly match a key recorded in the query model whitelist, and the rendered SQL expression must come from the CLI-generated sortable dictionary. Sort direction is limited to `asc` or `desc`.

### Included Responsibilities

- Use generated or loaded query source objects instead of `execute(sql: string, ...)` style public APIs.
- Carry SQL path, source SQL, query model analysis, and dialect binding metadata together.
- Verify source hash before using named-parameter or safe-sort metadata.
- Treat the query model safe-sort dictionary as the maximum runtime sort surface.
- Require safe-sort keys to match the query model whitelist exactly.
- Keep the raw string passed to the underlying DB driver inside the adapter implementation.
- Require generated SQL client and executor scaffolds to route runtime execution through the same file-backed or query-model-backed boundary.
- Treat scaffolded low-level connection helpers as non-execution helpers unless they also provide the file-backed or query-model-backed execution guard.

### Excluded Responsibilities

- Accepting arbitrary SQL strings as application-facing runtime input.
- Runtime dynamic SQL construction for filters, joins, projections, or business clauses.
- Sanitizing user-provided SQL fragments.
- Treating safe sort as an arbitrary ORDER BY string feature.
- Treating generated starter code as outside the concept check surface after it is written into the consuming project.

## Query Model Metadata Contract

### Definition

Query model metadata is the development-time SQL analysis record that lets Ashiba support bounded SQL handling without turning generated application code into an Ashiba runtime.

Runtime Zero can still support advanced SQL handling when the required SQL structure is analyzed during development and recorded beside the reviewed SQL. That recorded analysis is part of the query contract, not a loose cache.

### Why It Exists

Some SQL behavior needs structural knowledge that would normally require runtime parsing or dynamic SQL construction. Ashiba prefers to move that structural work to development time by using tested AST analysis and generating metadata that runtime code can verify.

That only remains safe when the SQL text and the recorded metadata describe the same query. If they drift apart, the metadata becomes misleading. Therefore source hashes or equivalent drift checks are mandatory before runtime code uses metadata that affects SQL handling.

### Included Responsibilities

- Treat SQL analysis metadata as part of the query contract.
- Generate structural metadata from development-time SQL analysis rather than runtime SQL parsing when possible.
- Keep metadata tied to the reviewed SQL path and source SQL.
- Verify source hashes or equivalent drift evidence before metadata is used for runtime SQL handling.
- Fail clearly when metadata is missing, stale, or inconsistent with the SQL text.
- Apply the same drift principle to composed metadata-driven behavior, such as binding, safe sort, and optional condition compression.

### Excluded Responsibilities

- Treating query model metadata as an optional runtime cache.
- Using stale analysis because the SQL string still parses or appears similar.
- Runtime AST parsing as the default way to recover missing metadata.
- Silent fallback from metadata-backed behavior to ad hoc SQL scanning.
- Hiding Ashiba-only markers inside SQL to avoid metadata generation.

## Thin Driver Adapter

### Definition

A thin driver adapter is allowed because raw database drivers are not enough for Ashiba's goals.

### Why It Exists

Named parameters, parameter checks, safe sort profiles, optional condition compression, and logger-ready execution events need a small DB driver wrapper seam without becoming an ORM.

Driver adapters should stay runtime-light. They should avoid runtime SQL AST parsing when development-time query metadata can make the behavior safe. Runtime AST parsing is not part of the current driver path. If metadata cannot solve a required SQL shape later, adding runtime AST parsing requires an explicit human design decision because it adds execution overhead and couples production behavior to parser defects.

Wrapper package names must make the wrapped library or executable explicit. Ashiba may support multiple PostgreSQL drivers or tools, so package names should not stop at the DBMS family. For example, the `pg` wrapper is `@ashiba/driver-adapter-pg`, while the PostgreSQL DDL pull helper backed by `pg_dump` is `@ashiba/ddl-pull-pg-dump`.

### Included Responsibilities

- Named parameter compilation.
- Parameter contract checks.
- Driver placeholder style.
- Ordered parameter name metadata.
- Safe sort profile interface and query-model-gated ORDER BY rendering as bounded driver-owned SQL handling.
- Optional condition compression as explicit query-model-gated SSSQL branch removal.
- Logger-ready event types.
- Masking policy types.
- Row result normalization contracts.
- Runtime-light execution that prefers CLI-generated query metadata over runtime AST parsing.
- Rejection of unsafe driver-side SQL handling when required metadata is missing or stale.
- File-backed or generated query source objects as the application-facing execution input.
- Package naming that includes the wrapped driver library or executable.

### Excluded Responsibilities

- ORM entities.
- Business SQL generation.
- Query DSL ownership.
- Relation magic.
- Runtime model tracking.
- Logger implementation.
- Transaction management policy.
- Runtime AST parsing as the default mechanism for SQL understanding.
- Arbitrary runtime SQL string execution as the application-facing boundary.
- Proprietary SQL comments, replacement markers, or template directives.

### Notes

Ashiba may scaffold feature code that shows transaction-shaped usage, following the `ztd-cli` baseline where useful. After generation, that code is application-owned and may be changed by humans or AI agents. Ashiba does not keep managing transaction boundaries.

## Named Parameter Binding

### Definition

Named parameter binding records SQL parameters such as `:userId` or `@userId` as development-time query model metadata, including the dialect-specific compiled SQL and ordered parameter names.

The preferred path is consistent with safe sort: `@ashiba/cli` records development-time binding metadata in the query model, including source SQL hash, dialect-specific compiled SQL, ordered parameter names, and any dialect-specific insertion offsets needed to compose other metadata-driven features. Runtime driver adapters should use that metadata when the hash matches and reject missing or stale metadata instead of rescanning SQL.

### Why It Exists

SQL must be maintainable, so Ashiba SQL should use named parameters instead of positional placeholders in source SQL. Readable names are easier to review, edit, and debug, but drivers often require positional values. The CLI owns metadata generation, and the DB driver wrapper owns runtime value mapping from reviewed metadata.

### Included Responsibilities

- Support named parameter source formats such as `:name` and `@name`.
- Generate dialect-specific binding metadata in the query model at development time.
- Verify source SQL hash before using precomputed binding metadata.
- When multiple metadata-driven rewrites compose, store positions for the generated dialect SQL as metadata rather than recalculating offsets at runtime.
- Fail before execution when query model binding metadata is absent or stale.
- Preserve `orderedNames`.
- Avoid rewriting strings, quoted identifiers, comments, and casts.

### Excluded Responsibilities

- Business SQL generation.
- Runtime model binding.
- Making application SQL use driver-specific positional placeholders as the source format.
- Runtime AST parsing for parameter binding.
- Runtime lexical fallback for parameter binding when query model binding metadata is missing.
- Using stale query model binding metadata.

## Safe Sort Profile

### Definition

Safe sort profiles define whitelisted sort keys and directions for DB driver wrapper use. Source SQL stays ordinary SQL: it must not contain Ashiba-only comments, replacement markers, or template directives. The preferred design avoids runtime AST parsing. The wrapper may render ORDER BY only when it is given CLI-generated query model metadata, a source SQL hash, a reviewed insertion plan, and a sortable dictionary.

### Why It Exists

Dynamic sorting is common, but raw user-provided ORDER BY strings are unsafe. A narrow sort profile plus CLI-generated query model analysis can define the sortable surface. The query model records where to insert sort SQL and whether an existing top-level `ORDER BY` requires comma append mode. This keeps runtime behavior deterministic without embedding Ashiba-only notation in SQL.

Runtime AST parsing is outside the current driver path. It adds overhead and couples production behavior to the AST parser's runtime correctness. Ashiba should generate stable insertion metadata at development time; the runtime driver should only verify metadata freshness, validate the sort input, and perform the bounded splice. If metadata cannot describe a SQL shape, the driver should reject it with guidance instead of parsing the SQL at runtime.

The sortable dictionary maps public sort names to reviewed SQL expressions. For example, `select a.user_id as id from users a` can expose `id` as a sort key whose SQL expression is `a.user_id`, so sorting by the DTO-facing name still renders `order by a.user_id`.

The query model sortable dictionary is the maximum runtime sort surface. Runtime options may refine presentation details such as default direction, but they must not replace the SQL expression recorded in the CLI-generated metadata.

Sort key matching is exact. A runtime sort request must use a key that exactly matches the query model whitelist; case folding, fuzzy matching, alias guessing, raw column expressions, and arbitrary ORDER BY fragments are rejected.

Root compound SELECT queries such as `UNION`, `INTERSECT`, or `EXCEPT` are intentionally conservative. The query model should record this as development-time metadata, such as `rootQueryShape: "compound-select"`, so the runtime driver does not need to inspect SQL text. Their stable sortable surface can be ambiguous, and Ashiba must not rewrite them into a different query shape. If safe sort is needed for a root compound query, the query should be written as an explicit subquery that exposes stable sortable columns, and Ashiba should report that next action instead of rewriting automatically.

### Included Responsibilities

- Allow only known sort keys.
- Require the requested sort key to exactly match a key in the query model whitelist.
- Allow only `asc` or `desc`.
- Require CLI-generated query model analysis for safe sort execution.
- Prefer development-time insertion metadata plus source SQL hash over runtime AST parsing.
- Reject sort input when query model metadata is missing or stale.
- Reject sort input unless the query model says the SQL parsed successfully and is a SELECT.
- Reject sort input when the ORDER BY insertion position is unresolved.
- Reject runtime sort profile SQL expressions that do not match the query model sortable dictionary.
- Reject root compound SELECT safe sort based on query model shape metadata, with guidance to wrap the compound query in an explicit subquery.
- Insert a new `ORDER BY` or append with a comma according to query model metadata.
- Keep ORDER BY rendering in the DB driver wrapper responsibility.
- Support dialect-aware identifier handling later.
- Never concatenate raw user text.

### Excluded Responsibilities

- Arbitrary SQL string assembly from user input.
- Inferring the correct ORDER BY insertion point from raw SQL at runtime.
- Adding runtime AST parsing as the default driver behavior.
- Detecting valid sortable columns from SQL text at runtime.
- Appending to an existing ORDER BY clause without explicit query model comma-mode metadata.
- Guessing an ORDER BY insertion point at runtime.
- Proprietary SQL comments, replacement markers, or template directives.
- ORM query planning.
- General SQL transformation or business query generation.

### Related Concepts

- `Thin Driver Adapter`
- `No Query DSL Ceremony`
- `CLI No Hidden SQL Rewrite`
- `Scalar Expansion`

## Optional Condition Compression

### Definition

Optional condition compression is explicit driver-owned SSSQL branch removal. Source SQL remains ordinary SQL with named parameters and supported optional-condition branches such as `(:status is null or status = :status)`. The SQL file must not contain Ashiba-only comments, replacement markers, or template directives.

The preferred path avoids runtime AST parsing. `@ashiba/cli` records supported optional-condition source ranges, dialect-specific compiled removal ranges, and the source SQL hash in the query model. Scaffolded query boundaries should generate this metadata by default because SQL-only edits are expected and metadata can be refreshed explicitly. The runtime driver may remove a recorded branch only when compression is explicitly enabled for that execution and the controlling parameter is explicitly present with `null` or `undefined`.

### Why It Exists

Optional filters are useful, but building WHERE clauses dynamically is a common SQL injection and debuggability risk. SSSQL keeps the full query visible and directly runnable, while metadata-backed compression removes absent optional branches without letting runtime code invent SQL structure.

This is intentionally optional. When compression is not enabled, the SQL executes exactly as authored. When compression is enabled, missing or stale metadata is an error, not a reason to scan SQL at runtime or silently fall back.

SSSQL authoring is a development-time SQL-file maintenance workflow, not a runtime SQL-string execution path. It may take authoring inputs such as `add user_id` or an EXISTS subquery and rewrite the SQL file to include the optional branch. Every successful SSSQL rewrite must refresh the query metadata used by runtime compression.

### Included Responsibilities

- Require explicit runtime opt-in for compression.
- Generate optional-condition compression metadata during query scaffolding and model generation.
- Provide an explicit metadata refresh command for SQL-only edits.
- Use CLI-generated query model metadata for removable branch ranges.
- Verify source SQL hash and dialect binding metadata before compression.
- Remove only branches recorded as supported during development.
- Treat `null` and `undefined` as absent only when the controlling parameter key is explicitly present.
- Compose with named-parameter binding and safe sort by using dialect-specific compiled ranges.
- Reject missing, stale, ambiguous, or overlapping metadata before execution.

### Excluded Responsibilities

- Runtime AST parsing as the default mechanism.
- Runtime discovery of optional branches from raw SQL text.
- Ashiba-only SQL markers or replacement comments.
- Free-form SQL condition assembly from user input.
- Silent fallback when metadata is missing or stale.

### Related Concepts

- `Named Parameter Binding`
- `Safe Sort Profile`
- `Thin Driver Adapter`

## Test Lanes

### Definition

Ashiba supports two unit-test lanes: traditional tests and Zero Table Dependency tests. The recommended lane depends on what the test is proving: mapper tests should prefer Zero Table Dependency, while performance tests should prefer traditional DB-backed tests.

### Why It Exists

Mapper tests need deterministic SQL-to-DTO contract checks without depending on unrelated table state. Zero Table Dependency fits that purpose, and the generated ZTD harness may share expensive connection resources inside one mapper-test run to preserve fast feedback. Connection open/close cost can dominate small DB-backed mapper checks; if the harness reconnects for every mapping case, it weakens the main ZTD benefit of fast repetition without migrations.

Performance tests need realistic database execution, query planning, indexes, data volume, timing behavior, and physical state lifecycle, so traditional DB-backed tests are the appropriate default and should not silently share mutable state unless the application test explicitly owns that policy. Traditional tests may share a pool when the test harness explicitly scopes setup, cleanup, transactions, and state ownership, but Ashiba should not assume that sharing is safe by default.

### Included Responsibilities

- Support traditional tests.
- Support Zero Table Dependency tests.
- Recommend Zero Table Dependency for mapper tests.
- Recommend traditional DB-backed tests for performance tests.
- Keep mapper-test connection reuse explicit and local to the ZTD harness.
- Avoid per-case connection open/close in generated ZTD mapper harnesses when a scoped shared connection can preserve deterministic isolation.
- Keep traditional/performance state lifecycle application-owned.
- Require traditional/performance connection sharing to be explicit about setup, cleanup, and mutable state ownership.
- Keep ZTD as a test technique, not the Ashiba product identity.

### Excluded Responsibilities

- Treating Zero Table Dependency as the only supported test style.
- Using Zero Table Dependency as the default for performance measurement.
- Treating traditional DB-backed tests as required for every mapper contract.
- Silently sharing mutable traditional/performance test state across cases.
- Reconnecting per generated ZTD mapper case when a scoped harness-level connection is sufficient.

## RFBA

### Definition

RFBA means Review-First Boundary Architecture. Ashiba separates files by review boundary, not by technical layer.

The primary boundary is VSA-style: a feature owns the public surface for one vertical slice, and query boundaries live under that feature. SQL, mapper contracts, execution contracts, and tests should stay close enough that a reviewer can inspect one behavior without jumping across repository-wide technical layers.

The exact target of a review is partly subjective. Ashiba does not try to solve that with an abstract rulebook. It fixes the practical review grain through scaffolding: feature files, query files, contracts, and tests are placed where a reviewer can repeatedly inspect the same kind of boundary.

### Why It Exists

Ashiba is a scaffolder for visible SQL and editable generated code. Reviewers usually ask which feature behavior is changing and which SQL that behavior runs. RFBA keeps the files arranged around that review question.

VSA-style boundaries also reduce the need for ORM concepts such as relation loading, lazy loading, and unit-of-work tracking. A feature can explicitly fetch the data it needs and decide its own transaction shape in application-owned code.

Features are not limited to one query. Adding another query boundary to an existing feature is a supported flow, because generated code is meant to be edited, extended, tested, and reviewed after the initial scaffold.

Feature boundaries may be subgrouped under the feature root when that matches the review responsibility, such as a write-side subgroup under a larger feature area. Imports that cross canonical roots or shared seams should use root-stable aliases instead of depth-sensitive relative paths, so moving a boundary deeper does not rewrite unrelated import depth.

### Included Responsibilities

- Feature-owned review boundaries.
- Query-local review boundaries under the feature.
- Multiple query boundaries inside one feature when the behavior needs them.
- Subgrouped feature boundaries under the feature root.
- Root-stable import aliases for cross-root or shared-seam references.
- Co-location of SQL, mapper contracts, execution contracts, and query-local verification.
- RFBA inspection commands that report discovered feature/query review boundaries.
- VSA-style separation by behavior or feature.

### Excluded Responsibilities

- Splitting generated application code by technical layer such as repository, service, DTO, mapper, and model folders as the primary organization.
- Making RFBA an ORM runtime architecture.
- Owning business transaction policy after scaffolding.
- Hiding SQL behind a layered data access abstraction.

## Query Boundary

### Definition

A query boundary is the feature-local access boundary for one named SQL behavior. It keeps SQL, query identity, mapping contract, execution contract, DTO-shaped result usage, and query-local verification together.

Application programmers should write feature code against the DTO or mapped result contract exposed by the query boundary, not against untyped raw database rows. The query boundary is where visible SQL becomes a typed application-facing contract.

### Why It Exists

SQL review and verification are easier when the query has a clear local public surface.

Type safety becomes useful only after SQL results are mapped into a stable contract that application code can depend on. Query boundaries make that contract visible and testable, while keeping the SQL close enough for review.

Queries also need stable names or IDs so debugging, drift checks, log traces, performance evidence, and AI-oriented error messages can point to the exact SQL behavior that ran.

### Included Responsibilities

- Stable query ID or query name.
- Query-local `query.ts`.
- SQL file.
- DTO or mapped result contract exposed to feature code.
- Parameter contract.
- Execution contract that accepts an application-owned executor or driver seam.
- Query-local tests.
- Generated analysis and test plans.
- Log/trace identity such as query ID and SQL path.

### Excluded Responsibilities

- Parent feature orchestration.
- Application transaction boundaries.
- Runtime relation loading, lazy loading, or unit-of-work tracking.
- Hidden dynamic SQL building for business conditions.
- Replacing the SQL file with a query DSL.

## Feature Boundary

### Definition

A feature boundary is the feature-owned boundary for SQL, QuerySpec, orchestration entrypoint, and feature-local verification.

A feature may contain one or more query boundaries. The feature is the reviewable behavior unit; each query boundary is a named SQL access point inside that behavior.

Feature boundaries can be nested below the feature root when the review unit needs a subgroup. The subgroup still owns a `boundary.ts` entrypoint and should not depend on fragile relative import depth for shared support.

### Why It Exists

Review responsibility and public surface should stay visible as features grow.

Feature boundaries follow RFBA and VSA-style organization: files are separated by reviewable behavior rather than by technical layer. A feature may contain DTOs, SQL, mapper contracts, execution contracts, and tests together when that keeps the review boundary coherent.

The scaffold fixes a consistent review grain for code review and AI review. It should support adding query boundaries to an existing feature instead of forcing a new feature for every SQL statement.

Stable root aliases are a scaffold support mechanism, not a replacement for RFBA. They are used where relative imports would make boundary movement expensive, especially shared feature seams and app-level test support.

### Included Responsibilities

- Feature public boundary.
- Feature-local query boundaries.
- Query additions to an existing feature boundary.
- Subgrouped boundaries with stable root-based shared imports.
- Feature-level tests.
- Documentation for the feature.
- VSA-style organization by behavior.

### Excluded Responsibilities

- Cross-feature private imports.
- Hidden framework-owned routing.
- Primary organization by technical layer.

### Related Concepts

- `Query Boundary`
- `RFBA`
- `Editable Generated Code`
- `Visible SQL`

## Pipeline Expansion

### Definition

Pipeline expansion turns large SQL into reviewable and executable pipeline steps without hiding SQL.

### Why It Exists

Large SQL often needs explainable slices, dependency views, and materialization planning while remaining SQL-first.

This intentionally lives outside the core `@ashiba/cli` Runtime Zero path. It is useful, but it is an extension capability because pipeline expansion changes how SQL is shaped for review or execution.

### Included Responsibilities

- CTE dependency analysis.
- Query pipeline plan.
- Materialization plan.
- Debuggable SQL slices.

### Excluded Responsibilities

- ORM query planning.
- Hidden performance tuning.
- Business SQL generation.

### Related Concepts

## Scalar Expansion

### Definition

Scalar expansion is a planned migration from the `rawsql-ts` / `ztd-cli` SSSQL optional-condition tooling. It supports SQL-first optional filter authoring and maintenance without query DSL ceremony.

### Why It Exists

Applications need optional conditions, and `rawsql-ts` already has a useful SQL-first baseline. Ashiba should migrate that baseline as an extension capability while preserving the principle that SQL remains visible, reviewable, and debuggable. It is not standard `@ashiba/cli` behavior because scalar expansion belongs outside the core CLI Runtime Zero path.

### Included Responsibilities

- SSSQL optional condition listing.
- Add SSSQL optional condition branches.
- Refresh existing optional branches and generated query metadata.
- Remove supported optional branches.
- Preserve existing SSSQL behavior from the migration source unless it conflicts with Ashiba's runtime boundaries.

### Excluded Responsibilities

- Query DSL ownership.
- Business query generation.
- Unbounded runtime dynamic SQL rewriting outside authored SSSQL behavior.
- Redefining core `@ashiba/cli` as a SQL transformation runtime.

### Related Concepts

- `No Query DSL Ceremony`
- `CLI No Hidden SQL Rewrite`
- `Safe Sort Profile`
- `Pipeline Expansion`

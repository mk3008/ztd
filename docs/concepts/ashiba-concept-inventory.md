# Ashiba Concept Inventory

## 1. Product Promise

Show me the SQL.

Ashiba handles the boring parts.

Ashiba keeps SQL visible.
Ashiba scaffolds DTO definitions, mappers, query ID numbering, tests, migrations, sqlgrep, and impact analysis.
Ashiba keeps query logic as SQL that can be run in a SQL client for debugging.
Ashiba source SQL must remain compatible SQL. Ashiba must not require proprietary comments, replacement markers, template directives, or other Ashiba-only SQL notation; if special Ashiba notation is required for the SQL to work, it is not SQL-compatible.
Ashiba uses named parameters such as `:name` or `@name` for SQL maintainability.
Named parameter binding should use CLI-generated query model metadata: source SQL hash, dialect-specific compiled SQL, ordered parameter names, and dialect-specific insertion offsets needed by composed features such as safe sort. Runtime adapters should reject missing or stale binding metadata instead of rescanning SQL.
Runtime execution boundaries should accept file-backed or generated query source objects, not arbitrary SQL strings. The source object carries SQL text, SQL path, and matching query model metadata; the raw SQL string passed to the underlying DB driver is an internal adapter detail. This is a scaffolded/generated-code requirement as well as a library requirement: SQL clients, executors, starter adapters, and connection seams generated by `@ashiba/cli` are part of the concept check surface.
Runtime Zero can still support bounded SQL handling when development-time AST analysis records the required query model metadata. That metadata is part of the SQL contract: if the SQL text and the recorded analysis drift apart, Ashiba must detect the mismatch through source hashes or equivalent drift checks before runtime code uses the metadata.
Ashiba CLI-generated code must not hide dynamic SQL rewriting. Safe sort is bounded driver-owned SQL handling based on a whitelisted sort profile and CLI-generated query model analysis, not Ashiba-only markers in SQL text. ORDER BY insertion is represented by development-time metadata: insertion position, new-clause or comma-append mode, and a sortable dictionary from public sort names to reviewed SQL expressions. Runtime AST parsing is not part of the driver path; unsupported SQL shapes should fail with guidance instead of being parsed at runtime.
Safe sort keys must exactly match the query model whitelist. No case folding, fuzzy matching, alias guessing, raw column expression, or free-form ORDER BY fragment is accepted as a sort key.
Optional SSSQL condition compression is allowed only as explicit driver-owned behavior backed by CLI-generated query model metadata. It must stay optional, require no Ashiba-only SQL notation, verify source hash, and remove only supported optional-condition ranges recorded during development. SSSQL authoring commands may accept development-time inputs such as column names or EXISTS subquery text, but their output is an updated SQL file plus refreshed query metadata; this is not a runtime SQL-string execution boundary.
Root compound SELECT safe sort, such as `UNION`, `INTERSECT`, or `EXCEPT`, should be represented in development-time query model metadata and fail at runtime with guidance to wrap the compound query in an explicit subquery when a stable sortable surface is needed; Ashiba must not silently rewrite or re-parse the query shape.
Ashiba does not add an ORM runtime.
Ashiba uses mapper tests, DB-backed integration tests, and drift checks to protect generated contracts.
Ashiba supports traditional and Zero Table Dependency test lanes. Mapper tests prefer ZTD; performance tests prefer traditional DB-backed tests.
Performance tuning is a manual traditional DB-backed evidence loop. Ashiba records representative row counts, response-time requirements, timeout status, execution-plan evidence, timings, and next actions. AI or humans decide candidate indexes and SQL changes; candidate indexes are sandbox-only until promoted into `db/ddl`.
Ashiba does not runtime-validate every DB result row.
Ashiba's precise claim is Ashiba Runtime Zero in application code generated by `@ashiba/cli`. The older phrase "runtime-free" is only a shorthand for that claim. CLI-generated application code avoids Ashiba CLI/runtime library dependency in application runtime, avoids forced security updates for an unused Ashiba runtime dependency, and lets teams patch generated code immediately without waiting for an upstream library release.
Ashiba compares two DDL inputs to generate migration DDL and risk info, without owning DB connections or apply.
Ashiba targets PostgreSQL, MySQL, and SQL Server; initial implementation starts with PostgreSQL.
Ashiba is not a PostgreSQL-only product. PostgreSQL may be used for the first starter and first production adapter, but product vocabulary, configuration names, environment variables, and package boundaries should stay DBMS-neutral unless a package explicitly wraps a PostgreSQL-specific driver or tool.
Ashiba uses thin driver adapters only where raw drivers are not enough; driver adapters and extension packages may have runtime dependencies. Driver adapters should still stay runtime-light: runtime AST parsing is not part of the current driver path. Unsupported SQL shapes should fail with guidance until development-time query metadata can describe them safely.
Logger-ready execution is also checked against scaffolded/generated application code. If generated code is presented as the application SQL execution path, it must preserve query ID/path, masked parameter, timing, row-count, and error event seams through a driver wrapper or clearly delegate to one. Low-level generated pool helpers are acceptable only as connection helpers, not as the scaffolded SQL client when they bypass those events.
Ashiba CLI should cover practical development support often expected from ORMs, excluding database drivers, through scaffolding, generated code, tests, drift checks, migration review, and query analysis instead of a runtime ORM layer.
Ashiba supports human-oriented and AI-oriented error output modes. Both modes must include the error cause and a next action or repair hint; AI-oriented output should be structured enough for agents, while human-oriented output should stay concise and actionable.
Ashiba public exported functions must have JSDoc. Ashiba CLI commands must expose help before execution; help may be split into human-oriented and AI-oriented forms when that makes the command contract safer for both readers.
Ashiba CLI commands that mutate files or external state must provide dry-run or equivalent preview behavior. Read-only inspection commands do not need dry-run because they are already observational.

## 2. Core Concepts

- `Ashiba`: SQL scaffolding product for TypeScript applications, with Ashiba Runtime Zero in application code generated by `@ashiba/cli`.
- `Visible SQL`: SQL remains readable, reviewable, editable, executable, searchable, and free of Ashiba-only SQL notation.
- `Named Parameter Binding`: source SQL uses named parameters such as `:name` or `@name`; the DB driver wrapper compiles them to driver-specific placeholders.
- `File-Backed Runtime SQL`: runtime execution boundaries use reviewed SQL files or generated query source objects with query model metadata, not arbitrary SQL string input.
- `Query Model Metadata Contract`: development-time SQL analysis metadata may enable Runtime Zero SQL handling only when it stays drift-checked against the source SQL.
- `Optional Condition Compression`: explicit driver-owned SSSQL branch removal using CLI-generated metadata; runtime does not parse SQL and does not infer removal ranges.
- `Boring Parts`: repetitive SQL-adjacent work such as DTO definitions, mappers, query ID numbering, tests, migrations, sqlgrep, impact analysis, and driver binding mechanics.
- `Ashiba Runtime Zero`: CLI-generated native TypeScript application code that does not require Ashiba CLI/runtime libraries at runtime; database drivers, driver adapters, and extensions are outside this claim.
- `No Query DSL Ceremony`: SQL must remain SQL and directly runnable in a SQL client for debugging.
- `CLI No Hidden SQL Rewrite`: the `@ashiba/cli` Runtime Zero path must not hide dynamic SQL rewriting in generated application code. Driver adapters and SQL-first extension packages may perform bounded SQL handling when their own package concept requires it, but source SQL must not depend on proprietary Ashiba syntax.
- `Editable Generated Code`: generated code is visible repository code, editable by humans and AI agents, not hidden behind `generate`, and protected by drift checks. Generated-owned metadata and human-editable code must not share one physical file; refreshed metadata belongs in generated metadata files.
- `Explicit Drift Recovery`: drift should be detected with clear failures, cause, and next action rather than hidden watch-mode regeneration.
- `RFBA`: Review-First Boundary Architecture; scaffolding fixes a repeatable VSA-style feature/query review grain instead of splitting by technical layers.
- `Mapper-Tested Type Safety`: DTO and mapper type safety is guaranteed by mapper tests and DB-backed integration tests, not runtime result-row validation.
- `Test Lanes`: traditional and Zero Table Dependency test lanes; mapper tests prefer ZTD, performance tests prefer traditional DB-backed tests.
- `Performance Tuning Session`: manual traditional DB-backed evidence loop; Ashiba records requirements and measurement evidence, while accepted indexes must be promoted into `db/ddl`.
- `Generated Mapping Tests`: library-owned ZTD cases for DB/driver type, nullable output, boundary value, and observable generated/default value mapping; human-owned logic cases stay separate.
- `Migration Query Generation`: CLI-owned DDL comparison that emits migration DDL and risk info without connecting to or mutating a database.
- `Error Output Modes`: human-oriented and AI-oriented error formats selected by execution parameter, with cause and suggested next action or repair hint in both modes.
- `Public API and Help Surface`: public exported functions require JSDoc, and CLI commands require help surfaces; AI-oriented help may be added when a structured command contract is safer.
- `CLI Dry Run`: mutating CLI commands must expose a preview mode that reports planned effects without writing files or changing external state.
- `Query Boundary`: feature-local named SQL access boundary for SQL, query ID, DTO/mapped result contract, parameter contract, execution contract, log trace identity, and query-local verification.
- `Feature Boundary`: feature-owned review boundary that may contain one or more query boundaries.

## 3. Responsibility Boundaries

Ashiba owns scaffolding, generated code layout, RFBA/VSA-style feature and query review boundaries, generated code that remains visible and editable by humans and AI agents, library-owned generated-folder unit-test schema files derived from DDL, DTO definitions, mapper scaffolding, query ID numbering, test scaffolding, drift detection, migration artifacts for review, migration query generation from DDL comparison, SQL search, SQL impact analysis, named parameter compilation, safe sort profile support, error output modes, and logger-ready event seams.

Review scope is partly subjective, so Ashiba should not depend on prose rules alone. Scaffolding owns the concrete review grain by creating feature boundaries and query boundaries that can be inspected repeatedly. Existing features may receive additional query boundaries when the behavior grows.

Feature boundaries may be subgrouped below the feature root when that better matches review responsibility. Scaffolded code should use root-stable aliases for cross-root or shared-seam imports such as feature shared support and test support, rather than making those imports depend on relative path depth.

Ashiba package public surfaces own their own JSDoc. Ashiba CLI commands own their help surface, including human-readable help and AI-oriented help when a structured form is useful. Help must be available before mutating or expensive operations run.

Ashiba mutating CLI commands own a dry-run or equivalent preview surface. The command should explain what files, SQL, or actions would be produced before writing them. Read-only report commands are already dry by nature and do not need a redundant flag.

Customer tests should model one customer's selected DBMS and wrapped driver at a time. A single product normally chooses one DBMS family for its runtime path, so PostgreSQL, MySQL, and SQL Server functional DB-backed customer flows should be verified independently rather than as one artificial all-drivers customer project. Customer tests are minimum functional tests, not installation tests: they must prove generated query metadata plus the selected driver adapter can execute real DB-backed CRUD and mapping checks for DB type, nullable output, boundary value, and generated/default value behavior.
Customer tests that need disposable SQL files, query contracts, tarballs, or generated metadata should place those artifacts under `tmp/` or another explicit temporary workspace. They must not normalize `src/sql` as an Ashiba project convention. Generated metadata names must be collision-free when multiple query contracts are produced in one directory.

Ashiba `init` may set up README files and ordinary project documentation, but Ashiba must not distribute `AGENTS.md`, `AGENT.md`, `SKILL.md`, skills, prompts, or other files that directly alter AI-agent behavior. Ashiba should guide AI agents through visible scaffolds, contracts, drift checks, and AI-oriented error messages instead.

Ashiba should not watch DDL files and automatically rewrite schema model or generated artifacts in the background. It should prefer explicit broken-state detection: tests or drift checks fail, explain what is stale, and tell the human or AI agent which explicit command to run. This keeps causality understandable and file diffs reviewable.

`@ashiba/cli` owns the Ashiba Runtime Zero promise for generated application code. Driver adapters and extension packages may have runtime dependencies when their package responsibility requires them.

Ashiba's product-level setup and test vocabulary should be DBMS-neutral. Use specific DBMS names only where the artifact is actually DBMS-specific, such as `@ashiba/driver-adapter-pg`, `@ashiba/ddl-pull-pg-dump`, or a PostgreSQL Compose starter service.

Ashiba setup should require explicit DBMS starter selection instead of silently choosing a database driver. `ashiba init` may create starter files for the selected DBMS, but package installation and `package.json` ownership belong to the consuming application.

Ashiba tooling packages may depend on `rawsql-ts` core for SQL AST analysis. That is a development-tooling dependency boundary, not a generated application runtime dependency. Generated DTOs, mappers, and feature code must not import Ashiba CLI/runtime libraries or `rawsql-ts` parser internals.

When Ashiba code runs only during development, SQL structural analysis should prefer the tested `rawsql-ts` AST over regular expressions or hand-written lexical parsing. Regex and lexical scans are acceptable for source offsets, generated TypeScript metadata extraction, parser-failure diagnostics, and path/string normalization, but table/column/clause/predicate/result-column understanding should migrate toward AST-backed extraction.

Development-time AST results that affect runtime behavior are not advisory notes. They become query model metadata contracts and must be kept in sync with the SQL text through source hashes or equivalent drift checks. If the SQL and metadata disagree, Ashiba should fail clearly before runtime metadata use rather than applying stale analysis.

Ashiba should not use implicit fallback when SQL structure cannot be understood. The `rawsql-ts` AST parser is a trusted and heavily tested dependency for dev-time SQL structure. If a dev-time feature needs regex to interpret SQL structure, first treat that as an Ashiba AST traversal gap or, when the parser itself cannot parse valid SQL, a `rawsql-ts` bug to investigate or report. The preferred response is to fail clearly, explain the unsupported SQL shape, and tell the human what should be fixed or explicitly requested. Silent repair, silent reinterpretation, or quiet fallback is rejected because Ashiba prefers explicit broken-state detection over quietly incorrect generated metadata.

The application owns business requirements, business SQL intent, transaction boundaries, connection lifecycle, migration apply execution, deployment, runtime logger implementation, and security policy for production logs. Ashiba may scaffold transaction-shaped starter code when creating features, but that code becomes editable application code and is not managed by Ashiba after generation. Ashiba scaffolding should not primarily split files by technical layers such as repository/service/model; reviewable feature/query behavior is the organizing boundary.

The `@ashiba/cli` Runtime Zero path must not hide SQL rewriting inside generated application code. This is a CLI/generated-code boundary, not a repository-wide ban on driver or extension behavior.

The driver adapter owns named parameter conversion to driver placeholders, parameter contract checks, unused parameter errors, bounded safe sort profile behavior using query model analysis, optional SSSQL condition compression when explicitly requested with query model metadata, and structured execution event generation for application-owned loggers. Its application-facing execution input should be a file-backed or generated query source object, not a bare SQL string. ORDER BY insertion and optional-condition removal use CLI-generated metadata and source hash verification. Runtime AST parsing is not part of the driver path; unsupported SQL shapes should fail with guidance instead of falling back to runtime parsing. CLI-generated SQL execution scaffolds must follow the same boundary; otherwise concept review has no library file to catch the violation.

The driver adapter does not own business SQL, ORM models, entities, relations, runtime change tracking, or query DSLs.

## 4. Runtime Policy

No ORM runtime. Generated native TypeScript application code from `@ashiba/cli`. Thin driver adapter only where needed.

Ashiba Runtime Zero means application code generated by `@ashiba/cli` can run without Ashiba CLI/runtime libraries after the CLI generates DTO definitions, mappers, and related code. It does not mean there is no database driver, no driver adapter, no extension runtime, no connection pool, or no application-owned transaction handling.

This matters because application runtime should not inherit forced security updates for an Ashiba runtime library that it does not need. If generated application code has a problem, teams can edit and patch that code directly instead of waiting for an Ashiba library release.

`@ashiba/cli` should still cover practical development support often expected from ORMs, excluding database drivers. It provides that support through scaffolded feature and query boundaries, DTOs, mappers, query IDs, tests, drift checks, migration review artifacts, and query analysis, not through relation loading, lazy loading, unit-of-work tracking, or a runtime entity model.

## 5. Driver Adapter Policy

Thin adapters are allowed where raw drivers are not enough. They may handle named parameters, parameter contract checks, safe sort profiles, logger-ready structured events, and row result normalization. They may own bounded SQL handling required by driver execution, but must not become ORM runtimes or business-query generators.

They must stay thin. They must not become ORM runtimes, query DSL owners, relation engines, unit-of-work managers, lazy loaders, or business SQL generators.

They must also keep SQL compatibility and runtime weight in view. They must not require Ashiba-only SQL notation. They should prefer CLI-generated query metadata over runtime AST parsing, and reject unsafe behavior when the metadata is missing or stale.

They should not expose `execute(sql: string, ...)` style application-facing APIs. Passing the final SQL string to the wrapped driver is allowed only inside the adapter after the query source, query model hash, named-parameter metadata, and safe-sort whitelist checks have passed.

## 6. Testing and Drift Policy

Runtime zero does not mean safety-free.

Safety moves from runtime abstraction to scaffolded tests. DDL, SQL, DTO types, and mappers must be checked during development. Mapper tests prefer Zero Table Dependency because they verify SQL-to-DTO contracts deterministically; their harness may share expensive connection resources within a mapper-test run for fast feedback. Per-case connection open/close can dominate small ZTD mapper checks and weaken the no-migration feedback loop. Performance tests prefer traditional DB-backed tests because they need real database behavior and application-owned physical state lifecycle; connection sharing in traditional tests must be explicit about setup, cleanup, transactions, and mutable state ownership. Runtime result-row type validation is not required by Ashiba when the mapper contract is proven. Generated code must be readable, editable, visible in the repository, and drift-detectable after human or AI edits.

## 7. Migration and Impact Policy

Migration artifacts are generated for review, not applied automatically. Migration query generation compares two DDL inputs and emits migration DDL plus risk info; it does not connect to the database. SQL impact analysis should expose table usage, column usage, observed SQL matching, and reviewable JSON output.

PostgreSQL DDL pull through `pg_dump` is represented by the `@ashiba/ddl-pull-pg-dump` helper package because the baseline already has `pg_dump`-related references. The package name is intentionally capability-plus-wrapper specific: `ddl-pull` names the Ashiba capability and `pg-dump` names the wrapped executable. A generic DDL pull package name would imply MySQL and SQL Server pull support too. It is not part of the initial migration query generation responsibility and should not live in the `pg` driver adapter. The reason is that DDL pull depends on live DB access and external DBMS tooling such as `pg_dump`, not on a TypeScript database driver API.

Current state: `ztd-cli` already contains many of these foundations. Ashiba needs migration, naming cleanup, and release packaging.

## 8. Extension Policy

Pipeline expansion and scalar expansion live outside the core `@ashiba/cli` Runtime Zero path, so they are extension capabilities rather than standard CLI behavior. Scalar expansion is planned for migration from the `rawsql-ts` / `ztd-cli` SSSQL optional-condition tooling. Extension packages may have runtime dependencies when needed, but should stay SQL-first, must not become ORM or query DSL packages, and must not redefine the core CLI as a hidden SQL rewriting runtime. Safe sort profile behavior belongs to the DB driver wrapper.

Prefer these SQL transform package names:

- `@ashiba/sql-transform-pipeline`
- `@ashiba/sql-transform-scalar`

Reserve, but do not implement yet:

- `@ashiba/extension-pipeline`
- `@ashiba/extension-scalar`

## 9. Package Naming Policy Summary

- CLI package: `@ashiba/cli`
- CLI command: `ashiba`
- Config command: `ashiba config`
- Production driver adapters: `@ashiba/driver-adapter-*`
- Testkit adapters: `@ashiba/testkit-adapter-*`

Driver and testkit adapter suffixes should follow wrapped driver names such as `pg`, `mysql2`, and `mssql`. This keeps PostgreSQL `pg` support distinct from possible future PostgreSQL driver adapters such as `postgres.js`.
- Driver-neutral SQL libraries: `@ashiba/sql-*`
- Dialects: `@ashiba/dialect-*`
- Extensions: `@ashiba/extension-*`, only after a plugin mechanism exists.

Do not use `sql-ashiba`.

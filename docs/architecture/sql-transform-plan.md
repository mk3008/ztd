# Ashiba SQL Transform Plan

## Goal

Plan Ashiba's future SQL transformation capabilities as extensions without turning core `@ashiba/cli` into a query DSL or hidden runtime SQL rewriter. Extension packages may have runtime dependencies; the Ashiba Runtime Zero promise belongs to application code generated by `@ashiba/cli`.

SQL remains SQL. Transforms may help maintain, slice, or scaffold SQL. They must not replace SQL with a library-owned query DSL, and transformed output must preserve a path to SQL that can be run in a SQL client for debugging.

Core `@ashiba/cli` should not hide dynamic SQL rewriting inside generated application code. Hidden rewriting makes the full query harder to understand, makes debugging harder, and increases injection risk. Pipeline and scalar expansion are useful but intentionally extension capabilities because they are not part of the CLI Runtime Zero path. Safe sort rendering and explicit optional-condition compression through reviewed query model metadata belong to the DB driver wrapper.

## Package Names

Use separate driver-neutral SQL transform packages, not the core CLI package:

- `@ashiba/sql-transform-pipeline`
- `@ashiba/sql-transform-scalar`

Do not create plugin-style extension packages yet.

Reserve for future:

- `@ashiba/extension-pipeline`
- `@ashiba/extension-scalar`

Only create `@ashiba/extension-*` packages when a plugin mechanism exists.

## Pipeline Expansion

Package: `@ashiba/sql-transform-pipeline`

Purpose: turn large SQL into reviewable and executable pipeline steps without hiding SQL.

Position: extension capability. It is useful for large SQL, but it is not standard core CLI behavior because it changes SQL shape.

Responsibilities:

- CTE dependency analysis.
- Query plan generation.
- Materialization plan.
- Debug SQL slices.
- Step-by-step execution plan.
- Stable output for AI review.

Non-responsibilities:

- Business SQL generation.
- ORM query planning.
- Automatic hidden performance tuning.

Existing `ztd-cli` references to inspect:

- `query outline`
- `query graph`
- `query slice`
- `query plan`

## Scalar Expansion

Package: `@ashiba/sql-transform-scalar`

Purpose: migrate the `rawsql-ts` / `ztd-cli` SSSQL optional-condition tooling into Ashiba as SQL-first optional-condition maintenance.

Position: extension capability. It is not standard core CLI behavior because scalar expansion is a form of SQL transformation.

Responsibilities:

- SSSQL optional condition listing.
- Add SSSQL optional condition branches.
- Refresh existing optional branches and generated query metadata.
- Remove supported optional branches.
- Allow development-time authoring inputs, including EXISTS subquery text, only when the output is written back to a SQL file.
- Preserve comments and SQL readability where possible.
- Preserve migration-source behavior unless it conflicts with Ashiba's runtime boundaries.

Non-responsibilities:

- Unbounded runtime dynamic SQL rewriting outside authored SSSQL behavior.
- Business condition assembly from user input.

Existing `ztd-cli` references to inspect:

- `query sssql list`
- `query sssql add`
- `query sssql refresh`
- `query sssql remove`
- `packages/core/src/transformers/SSSQLFilterBuilder.ts`
- `packages/core/src/transformers/PruneOptionalConditionBranches.ts`
- `packages/core/src/transformers/DynamicQueryBuilder.ts`

## Safe Sort Profile

Package: none planned as a transform package.

Purpose: safe SQL execution requires whitelisted sort profiles. This is a DB driver wrapper responsibility, not a transformer responsibility.

Status: low priority compared with CLI migration, but required before safe production execution support is complete.

Responsibilities:

- Sort profile schema.
- Allowed sort keys.
- Allowed directions.
- Runtime safe sort rendering only when ORDER BY insertion is represented by reviewed query model metadata: insertion position, order-by/comma mode, and sortable dictionary.
- No Ashiba-only SQL comments, replacement markers, or template directives.
- No raw user SQL concatenation.

Non-responsibilities:

- General SQL transformation.
- Query DSL ownership.

Source evidence to inspect:

- `packages/core/src/transformers/SqlSortInjector.ts`
- `packages/core/tests/transformers/SqlSortInjector.test.ts`

## Optional Condition Compression

Package: none planned as a transform package for runtime compression.

Purpose: allow authored SSSQL optional branches to be removed at execution time without runtime SQL parsing or Ashiba-only SQL syntax.

Position: DB driver wrapper capability. `@ashiba/cli` generates metadata during development; the runtime adapter uses it only when explicitly enabled. This differs from scalar expansion commands, which maintain or scaffold SQL files.

Responsibilities:

- CLI-generated source removal ranges and dialect compiled removal ranges.
- Source hash verification.
- Explicit runtime opt-in.
- Missing/stale metadata errors.
- Composition with named parameter binding and safe sort.

Non-responsibilities:

- Runtime optional-condition discovery from SQL text.
- General SQL transformation.
- Business condition assembly.

Do not implement all commands now.

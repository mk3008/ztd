# Ashiba Driver Adapter Plan

## Goal

Design the production driver adapter layer for Ashiba. This is not an ORM. It is a thin runtime safety and driver compatibility seam. Driver adapters may be runtime dependencies; the Ashiba Runtime Zero promise applies to application code generated by `@ashiba/cli`.

`@ashiba/driver-adapter-core` now exists as the shared runtime contract package for the first concrete adapter.

## Package Names

- Core: `@ashiba/driver-adapter-core`
- First driver: `@ashiba/driver-adapter-pg`

Future drivers:

- `@ashiba/driver-adapter-mysql2`
- `@ashiba/driver-adapter-mssql`
- `@ashiba/driver-adapter-postgres-js`

Target database families are PostgreSQL, MySQL, and SQL Server. Initial implementation starts with PostgreSQL.

## Core Responsibilities

Driver adapter core owns:

- Shared runtime metadata contracts.
- Parameter contract checks.
- Unused parameter errors.
- Safe sort profile interface.
- Logger-ready event types.
- Masking policy types.
- Row result normalization contracts.

## Driver-Specific Responsibilities

Driver-specific packages own:

- Adapting concrete driver Client or Pool.
- Dialect default placeholder style.
- Connection lifecycle integration points.

## Application Responsibilities

The application owns:

- Creating Pool or Client.
- Connection string.
- Transaction policy.
- Transaction management.
- Runtime logger implementation.
- Production masking policy.
- Business SQL.

## Must Not Implement

Do not implement ORM models, entities, relations, relation loading, lazy loading, runtime unit of work, dirty checking, query DSLs, business SQL builders, logger implementations, transaction management policy, or DDL pull.

Feature scaffolding may generate transaction-shaped starter code when that matches the `ztd-cli` baseline, but generated feature code is application-owned and may be changed freely. Ashiba does not keep managing it after scaffolding.

DDL pull should stay outside the PostgreSQL driver adapter. It depends on live DB access and external DBMS tooling such as `pg_dump`, rather than TypeScript driver execution semantics. Put it in a schema pull extension package instead.

## Named Parameters

Source SQL should use named parameters for maintainability. Both `:name` and `@name` source formats are acceptable. The DB driver wrapper owns conversion from named parameters to the concrete driver placeholder format.

Source SQL:

```sql
select
  id,
  email
from public.users
where id = :userId;
```

Alternative source SQL:

```sql
select
  id,
  email
from public.users
where id = @userId;
```

Postgres compiled SQL:

```sql
select
  id,
  email
from public.users
where id = $1;
```

Compiled metadata:

```txt
orderedNames: ["userId"]
```

Preferred runtime path:

1. `@ashiba/cli` writes query model metadata with source SQL hash, Postgres compiled SQL, and `orderedNames`.
2. The driver adapter verifies the source SQL hash.
3. If the hash matches, the driver uses the precomputed compiled SQL and maps values by `orderedNames`.
4. If metadata is absent, the driver fails before execution and asks the caller to regenerate the query model.

Missing and unused params must fail before execution. Runtime AST parsing and runtime SQL-scanning binders are not needed for named parameter binding.

## Safe Sort Profile

The DB driver wrapper owns bounded safe sort behavior. Do not accept raw user-provided ORDER BY strings, do not infer a sort insertion point from SQL text at runtime, and do not use this driver responsibility for general SQL transformation.

Use whitelisted sort profiles:

```ts
const userSortProfile = {
  createdAt: { sql: '"created_at"', defaultDirection: "desc" },
  name: { sql: '"name"', defaultDirection: "asc" }
} as const;
```

User input:

```ts
[
  { key: "createdAt", direction: "desc" },
  { key: "name", direction: "asc" }
]
```

Generated SQL fragment:

```sql
order by "created_at" desc, "name" asc
```

Source SQL must remain ordinary SQL and must not contain Ashiba-only comments, replacement markers, or template directives. If sort input is provided, the driver wrapper requires CLI-generated query model analysis proving that the SQL parsed successfully and is a SELECT. The query model must also carry safe-sort metadata: the insertion index, whether the splice starts a new `ORDER BY` clause or appends with a comma, and a sortable dictionary from public sort keys to reviewed SQL expressions.

Runtime AST parsing is not part of the driver path. `@ashiba/cli` generates source SQL hash metadata and reviewed ORDER BY insertion metadata during development. Runtime verifies metadata freshness, validates whitelisted sort input, and performs only a bounded splice. If the metadata approach cannot support a future SQL shape, the driver should reject that shape with guidance rather than parse SQL at runtime. Sort profiles must allow only known keys, allow only `asc` or `desc`, never concatenate raw user text, and support dialect-aware identifier handling later.

Root compound SELECT queries such as `UNION`, `INTERSECT`, or `EXCEPT` should be recorded by `@ashiba/cli` as query model shape metadata, for example `rootQueryShape: "compound-select"`. The runtime driver can then reject safe sort without parsing SQL text. It should reject unless the application writes an explicit subquery that exposes stable sortable columns. The driver adapter must not silently rewrite a root compound query into a subquery.

## Optional Condition Compression

The DB driver wrapper may own explicit SSSQL optional-condition compression when the query model contains CLI-generated removal metadata. This is not a general SQL transform package and not a runtime AST feature.

Rules:

- Compression is off by default and requires an explicit execution option.
- Source SQL remains ordinary SQL and must not contain Ashiba-only markers.
- Runtime uses source hash, source removal ranges, compiled removal ranges, and ordered parameter metadata generated by `@ashiba/cli`.
- Runtime removes only supported branches whose controlling parameter key is explicitly present with `null` or `undefined`.
- Missing, stale, ambiguous, or overlapping metadata fails before execution.
- Compression must compose with named parameter binding and safe sort without reparsing SQL at runtime.

## Logger-Ready Event Seam

Ashiba does not ship a logger. It exposes a structured observer seam, owned by the DB driver wrapper, so applications can connect their own logger.

Event data should support:

- `phase`
- `sqlId`
- `queryId`
- `sqlFile`
- `sqlPath`
- `dialect`
- `sourceSql` or `sourceSqlHash`
- `compiledSql` or `compiledSqlHash`
- `orderedNames`
- `maskedParams`
- `params` when explicitly unmasked
- `elapsedMs`
- `rowCount`
- `error summary`

Parameter values should be masked by default. Unmasked parameters are allowed only when explicitly requested by the application/operator.

Suggested masking policy:

```ts
export type AshibaMaskPolicy = "always" | "development" | "never";
```

Default: `always`.

The driver adapter must not depend on pino, winston, console logger, or OpenTelemetry SDK.

## Error Output Modes

Driver adapters and CLI commands should support human-oriented and AI-oriented error output modes. The mode is selected by execution parameter or command option. Errors should include the cause and, when safe and practical, a suggested next action.

## Suggested Core Types

```ts
export type AshibaMaskPolicy = "always" | "development" | "never";

export type AshibaSqlExecutionMetadata = {
  sqlId?: string;
  queryId?: string;
  sqlFile?: string;
  sqlPath?: string;
  dialect?: string;
};

export type AshibaSqlExecutionEvent = {
  phase: "start" | "end" | "error";
  metadata?: AshibaSqlExecutionMetadata;
  sourceSql?: string;
  compiledSql?: string;
  orderedNames?: readonly string[];
  maskedParams?: readonly unknown[];
  params?: readonly unknown[];
  elapsedMs?: number;
  rowCount?: number;
  error?: {
    name: string;
    message: string;
  };
};

export type AshibaSqlExecutionObserver = {
  emit(event: AshibaSqlExecutionEvent): void;
};
```

## Implementation Status

- Named-parameter metadata generation is integrated into `@ashiba/cli`; it emits compiled SQL and ordered names into the query model.
- `@ashiba/driver-adapter-pg` maps runtime parameter values from CLI-generated metadata and rejects missing or unused values before execution.
- `@ashiba/driver-adapter-core` has initial shared event, masking, error, and safe sort helpers.
- `@ashiba/driver-adapter-pg` has an initial `pg` compatible adapter with contract tests for query delegation, parameter checks, query-model-gated safe sort rendering from insertion metadata, stale metadata rejection, and observer events.
- Optional live PostgreSQL smoke exists and runs when `ASHIBA_TEST_DATABASE_URL` or `DATABASE_URL` is provided. Production readiness still requires running that smoke against a real PostgreSQL instance in release verification.
